# Dataset Registry ‚Äî project skill

Internal research dataset catalogue. **GitHub is the database for catalogue metadata.** Supabase handles auth, membership, and stars only.

Binding spec: [plan.md](plan.md). README: [README.md](README.md).

---

## Tech stack

| Layer | Choice | Notes |
|-------|--------|-------|
| Framework | **Next.js 16** App Router | `app/` routes, Route Handlers under `app/api/` |
| Runtime | **Node.js** on Vercel | Catalogue APIs use `export const runtime = "nodejs"` |
| UI | **React 19**, **Tailwind CSS 4**, **shadcn/ui** | Base UI via `@base-ui/react`; read `.agents/skills/shadcn/SKILL.md` before UI work |
| Design | Claude dark palette + terracotta accent | Tokens in `app/globals.css`; layout/visual rules in `.cursor/agents/claude-design.md` |
| Auth | **Supabase Auth** (GitHub OAuth) | Cookie sessions via `@supabase/ssr`; middleware refreshes session |
| Membership | **Supabase Postgres** | `catalogue_members`, `profiles`, `dataset_stars` ‚Äî not catalogue JSON |
| Catalogue data | **Private GitHub repo** | Fine-grained PAT + Contents API; **direct commits to `main`**, no PR workflow |
| Validation | **AJV** + `schema/dataset.schema.json` | Shared by API, forms, CI (`scripts/validate-datasets.mjs`) |
| Package manager | **pnpm 9** | `packageManager` in `package.json`; CI uses frozen lockfile |

**Out of scope:** Seekly apps/projects. Never deploy this repo to Seekly Vercel or Seekly Supabase.

## Architecture (two backends)

```text
Browser
  ‚Üí Next.js (Vercel)
      ‚Üí Supabase: session, catalogue_members, stars
      ‚Üí GitHub Contents API: datasets/*.json, datasets/*.md, config/classification-vocabulary.json
```

- **Supabase** answers: who is logged in? are they a member/admin? who starred what?
- **GitHub** answers: what datasets exist? what are their fields? what is the controlled vocabulary?

Never store catalogue metadata in Supabase tables. Never treat `generated/index.json` as source of truth when `GITHUB_*` env is set.

## Git as backend ‚Äî repo layout

```text
datasets/<id>.json          # one JSON file per dataset (canonical record)
datasets/<id>.md            # optional long Markdown description
config/classification-vocabulary.json   # admin-managed enums (modality, task, ‚Ä¶)
schema/dataset.schema.json  # JSON Schema
generated/index.json        # build artefact; regenerated by scripts + CI
scripts/generate-index.mjs
scripts/validate-datasets.mjs
lib/github/contents.ts      # listDirectory, getBlobFile, putBlobFile, deleteBlobFile
```

**Id rules:** slug `[a-z0-9-]+`; filename stem must match JSON `id` field (`lib/catalogue/path.ts`).

**Commit messages:** `catalogue: add dataset ‚Ä¶`, `catalogue: update dataset ‚Ä¶`, `catalogue: delete dataset ‚Ä¶`, `chore(catalogue): ‚Ä¶` for vocabulary.

**No PRs** for catalogue files ‚Äî Route Handlers commit straight to `GITHUB_DEFAULT_BRANCH` (default `main`).

## Read path (how the UI gets data)

1. **Live (production / dev with env):** `fetchCatalogueIndexLive()` lists `datasets/*.json` via GitHub API (`lib/catalogue/fetch-index-live.ts`). Cached ~30s with tag `catalogue-index-live`; client hooks call `GET /api/catalogue/index` (uncached) + `router.refresh()` after writes.

2. **Fallback:** `getCatalogueIndex()` reads bundled `generated/index.json` when `GITHUB_TOKEN` / `GITHUB_REPOSITORY` missing (offline dev).

3. **Detail/edit:** `getDatasetEntryServer()` prefers index, falls back to `GET /api/catalogue/datasets/[id]` (GitHub blob).

4. **Vocabulary:** `loadClassificationVocabularyLive()` from GitHub `config/classification-vocabulary.json`; mirrors to local disk on read/write; tag `classification-vocabulary-live`.

**Implication:** UI can show new datasets immediately after save without local `git pull`. Local checkout may lag until pull ‚Äî **admin UI commits to GitHub, not your working tree**.

## Write path (mutations)

All writes go through **authenticated Route Handlers** ‚Üí **`lib/github/contents.ts`**:

| Action | API | GitHub paths |
|--------|-----|--------------|
| Create/update dataset | `PUT /api/catalogue/datasets/[id]` | `datasets/<id>.json` |
| Long description | `PUT /api/catalogue/datasets/[id]/description` | `datasets/<id>.md` |
| Delete dataset | `DELETE /api/catalogue/datasets/[id]` | JSON + optional MD |
| Admin vocabulary | `POST/DELETE /api/admin/classification` | `config/classification-vocabulary.json` |

After PUT/DELETE dataset: `revalidateTag(CATALOGUE_INDEX_CACHE_TAG)`. After vocabulary change: `revalidateTag(CLASSIFICATION_VOCABULARY_CACHE_TAG)`.

**Ownership:** server stamps `created_by`, `created_by_user_id`, `created_by_email` on create from Supabase session ‚Äî browser never sets these.

**Auth gates** (`lib/catalogue/access.ts`):

- Read catalogue APIs: catalogue **member**
- Create dataset / edit own: member + owner or **admin**
- Admin members/vocabulary: **admin** only
- `CATALOGUE_E2E_BEARER`: local/non-prod smoke only; ignored when `VERCEL_ENV=production`

## Supabase (what lives there)

Migrations in `supabase/migrations/`:

- `catalogue_members` ‚Äî email allowlist, role (`member` | `admin`), display name
- `profiles` ‚Äî linked to auth users
- `dataset_stars` ‚Äî per-user starred dataset ids (not in Git)

Manage members at `/datasets/admin` (not env vars, except optional `CATALOGUE_BOOTSTRAP_ADMIN_EMAILS` for first admin).

**Required server env:** `SUPABASE_SERVICE_ROLE_KEY` for membership checks and star cleanup on delete.

## Local dev & CI

```bash
pnpm install
pnpm dev                    # localhost:3000
pnpm run validate:datasets  # all datasets/*.json vs schema
pnpm run generate:index       # refresh generated/index.json
pnpm build                    # index + validate + next build
```

**Env** (`.env.example` ‚Üí `.env.local`): `NEXT_PUBLIC_SUPABASE_*`, `SUPABASE_SERVICE_ROLE_KEY`, `GITHUB_TOKEN`, `GITHUB_REPOSITORY`, optional `GITHUB_DEFAULT_BRANCH`, optional `CATALOGUE_E2E_BEARER`.

**CI:** `.github/workflows/validate-datasets.yml` runs build on `main`. Vercel deploy is separate ‚Äî treat Actions green as health check for catalogue JSON.

## Git workflow pitfalls (read before push)

1. **Admin saves commit to GitHub immediately** ‚Äî local `main` falls behind. Before `git push`: `git pull --rebase origin main`.

2. **New dataset visible in app but missing locally** ‚Äî expected until `git pull`. App reads GitHub live; your checkout does not.

3. **Do not commit secrets** ‚Äî `GITHUB_TOKEN`, `SUPABASE_SERVICE_ROLE_KEY`, `CATALOGUE_E2E_BEARER` are server-only (never `NEXT_PUBLIC_*`).

4. **Do not hand-edit `generated/index.json` for production truth** ‚Äî run `pnpm run generate:index` or let build regenerate; runtime prefers GitHub when configured.

5. **Classification vocabulary** ‚Äî admin API writes GitHub; `mirrorVocabularyDiskIfStale` updates local `config/classification-vocabulary.json` on read/write. Still pull before push if others edited via admin.

## Agent checklist (catalogue changes)

When adding/editing catalogue behaviour:

- [ ] Validate against `schema/dataset.schema.json` and vocabulary if enum-backed
- [ ] Mutations only via Route Handlers ‚Üí `lib/github/contents.ts`
- [ ] Auth via `lib/catalogue/access.ts` / `editor-session.ts`
- [ ] Invalidate correct cache tag after GitHub write
- [ ] Client lists use live index hook or uncached API where stale UI was reported
- [ ] UI: shadcn + existing surface style constants
- [ ] Run `pnpm lint` and `pnpm build`
- [ ] Remind user to `git pull --rebase` if they will push after admin UI commits
